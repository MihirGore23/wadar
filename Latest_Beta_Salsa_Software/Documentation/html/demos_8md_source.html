1 # Salsa C Demonstration User Guide #
2 
3 Salsa software also comes with a variety of application demonstrations. Many of
4 these demo apps build on one another to help you understand how to create your
5 own radar application.
6 
7 Once you have Salsa installed on your BBB there will be a folder called FlatEarth.
8 Within that folder will be several sub-folders including:
9 + MatlabConnector
10 + Demos
11 + SalsaScope

49 <h2 id = "getting_started">Getting Started with C Development</h2>
50 
51 There are a number of C demonstration programs written for both the BeagleBone
52 Black (BBB) and Linux PC platforms.
53 
54 When installed, the C demonstration programs are located on the BBB in the
55 location:
56 @verbatim
57 /root/FlatEarth/Demos
58 @endverbatim
59 
60 This folder contains a number of sub-folders. Each cape has its *own* folder.
61 The Ancho and Cayenne folders contain demonstration code which runs on the BBB
62 platform. The Desktop_Ancho and Desktop_Cayenne folders contain demonstration
63 code which runs on a Linux PC. *The PC demos work in conjunction with appropriate
64 cape's BBB RadarServer demo app*. The Common folder contains some shared code
65 which works on both the BBB and Linux PC platforms.

100 To run and or edit a demonstration application, open a terminal, navigate into
101 each demo folder and then type 'make' to build it.
102 
103 The "hello world" for radar cape development starts with RadarDemo at the
104 lowest level of complexity. This demo is further enhanced in several iterations
105 in the following progression:
106 @verbatim
107 RadarDemo
108  └─ClutterDemo
109  └─RangingDemo
110  └─ZoneSense
111 @endverbatim
112 
113 This set of demos is intended to show how to build up complexity when writing
114 code for the radar capes in the C language. These demos are all similar as they
115 process individual radar frames without history; in other words, a radar frame
116 is captured, processed, then the loop repeats. These demos are mostly useful for
117 viewing data in a similar fashion as SalsaScope, but here the code is available
118 for inspection.
119 
120 The DopplerDemo project uses the pulse Doppler processing technique. Instead of
121 processing individual radar frames as they arrive, in this technique, a limited
122 history of past frames is also used. Doppler is particularly useful to detect
123 targets in motion. In addition to distance estimation, Doppler is also useful to
124 discover direction of travel. The ZoneSenseDoppler project builds upon the
125 DopplerDemo project, but adds in some additional complexity.
126 
127 Often, it is difficult to obtain realistic radar data at a developer's desk. The
128 FrameLogger demo folder contains two programs, the FrameLogger itself, and the
129 corresponding FrameViewer. These programs serve as basic examples on how a data
130 logging program could work. The FrameLogger program captures radar frames and
131 saves them to a binary file. The FrameViewer program reads these files back and
132 can display the data via Gnuplot. However, FrameViewer is *more* useful as a
133 template for a radar data post-processing application.
134 
135 The RadarServer demo is an example of a TCP server. It is used to generate data
136 for the Linux desktop PC demo apps. The set of Linux desktop apps use the CSFML
137 graphics API for radar signal visualization. To make a new demo, the user would
138 make a clone of one of the existing desktop demos, then implement the functions
139 declared that folder's radarClient.h header.

210 The Desktop_Ancho and Desktop_Cayenne folders contain demonstration applications
211 which run on a Linux desktop PC and communicate with the appropriate _RadarServer_
212 demonstration app. The server has identical functionality for each cape. The
213 desktop demos are nearly the same for each cape, but there are some small
214 differences, so use the appropriate cape's desktop demo.

216 @verbatim

225 
226 The _RadarClient_ sub-folder contains shared code that each of the demonstration
227 apps use. This code understands the protocol used by the _RadarServer_. The
228 radarClient.h header declares several functions. Each of these demonstration
229 apps then implements these functions in different ways depending on the
230 visualization and DSP.

233 all the DSP and viewing is handled by the local Linux PC.

341 <h2 id = "linux_pc_prep">Installing software for Linux PC Demos</h2>
342 
343 In order to use the Linux desktop demo apps, a number of steps are required in
344 advance. The first thing to do is determine the appropriate version of the
345 Debian packages to install. Open a terminal on the Linux desktop, and enter:
346 @verbatim
347 $ arch
348 @endverbatim
349 
350 A 32-bit Linux installation will respond with i686. A 64-bit Linux installation
351 will respond with x86_64. The two Debian packages are named using the format:
352 @verbatim
353 salsalib_<date>_<arch>.deb
354 salsalib-dev_<date>_<arch>.deb
355 
356 where...
357 
358 <arch> is i386 for 32-bit Linux PC
359 <arch> is amd64 for 64-bit Linux PC
360 @endverbatim
361 
362 Copy the appropriate .deb files to the Linux PC; they are located in the
363 "Installers\LinuxInstallers\PC" folder in the included USB stick.
364 
365 The Debian packages can then be installed using the command-line:
366 @verbatim
367 sudo dpkg -i [filename]
368 @endverbatim
369 
370 They can also be installed just by double-clicking the file's icon and then
371 clicking the 'install' button in the package manager window which opens. _If the
372 same Debian package has already been installed, the button will display as
373 'reinstall' instead_.
374 
375 The Linux desktop demonstrations use the SFML graphics API for visualization.
376 SFML is a lightweight, multi-platform, multi-language API.
377 
378 To install 'C' version of SFML on an Ubuntu system:
379 @verbatim
380 sudo apt-get update
381 sudo apt-get install libcsfml-dev
382 @endverbatim
383 
384 Next, copy the Demos/Common folder and then either the Demos/Desktop_Ancho or
385 Demos/Desktop_Cayenne folders to a location on the Linux desktop.

468 <h2 id = "configuring_radar">Configuring the Radar</h2>
469 
470 The Radarlib3 API provides mechanisms to control the radar IC by updating
471 variables, reading variables, performing actions, etc.
472 
473 For example, to set the _FrameStitch_ variable to 2 the code would look like:
474 @code
475 radarStatus = NVA_VarSetIntValue_ByName(radarHandle, "FrameStitch", 2);
476 if (radarStatus != NVA_SUCCESS)
477 {
478  NVA_GetErrorDescription(radarHandle, radarErrorMsg, sizeof(radarErrorMsg), radarHandle->errorHandle->lastError);
479  fprintf(stderr, "%s\n", radarErrorMsg);
480  return (1);
481 }
482 @endcode
483 
484 Radarlib3 variables are of type Value_t which is a _Union_ composed of an int
485 and float. _There is also a more generic call, NVA_VarSetValue_ByName(), but it
486 takes Value_t arguments. Using it is more complicated_.
487 
488 In general, most apps will require some customization of the Radarlib3 variables.
489 This same code block could be repeated for each integer type variable; there is
490 a similar function to set float variables.
491 
492 The Salsa radar helper API provides a mechanism configure the radar using JSON
493 files making the code much shorter. Each BeagleBone Black demonstration contains
494 two files, "stage1.json" and "stage2.json".
495 
496 In the JSON files, each setting is represented as a { "key" : value } pair. The
497 "key" is the string which represents the name of a Radarlib3 variable, and the
498 value is a number. The API will read the number as an _int_ or _float_ depending
499 on the variable name. If the value is "null" then that variable is treated as a
500 "read-only" value.
501 
502 The JSON configuration files are loaded in two stages. _There are a number of
503 Radarlib3 variables which only work after a timing measurement has been made_.
504 The example JSON files are organized to demonstrate this.
505 
506 ### stage1.json: ###
507 
508 @verbatim
509 [
510  {"DACMin" : 1000},
511  {"DACMax" : 7000},
512  {"DACStep" : 32},
513  {"Iterations" : 200},
514  {"PulsesPerStep" : 16},
515  {"FrameStitch" : 1}
516 ]
517 @endverbatim
518 
519 The first five settings in stage1.json effect the sampling circuitry. These
520 will typically be modified for specific applications. These are the primary
521 variables which effect the radar frame rate and signal to noise ratio.
522 
523 The next setting, 'FrameStitch' determines how many individual radar frames
524 are concatenated or "stitched" together. Since the frames are independent,
525 using this feature will reduce the overall framerate when the value is > 1.
526 
527 @note
528 On the Ancho (XeThru X2) each individual radar frame is composed of 256 samples
529 (or bins) spanning a distance of ~ 1 m. The Cayenne (XeThru X1) frame 512 samples
530 spanning ~ 2 m. The _FrameStitch_ value can be used to increase the overall frame
531 span at the expense of measurement time.
532 
533 In practical applications, the timing measurements should _always_ be executed
534 after the stage1 configuration. The stage2 configuration includes some additional
535 parameters which should be set only after the timing measurement.
536 
537 ### stage2.json: ###
538 
539 @verbatim
540 [
541  {"OffsetDistanceFromReference" : 0.1},
542  {"SampleDelayToReference" : 3.867e-9},
543  {"SamplesPerSecond" : null},
544  {"SamplersPerFrame" : null},
545  {"SampleDelay" : null}
546 ]
547 @endverbatim
548 
549 'OffsetDistanceFromReference' controls the distance (in meters) at which to
550 start sampling the radar from. Typically, the reference point is something
551 tangible like the surface of an antenna radome.
552 
553 @note
554 _OffsetDistanceFromReference_ and _FrameStitch_ can be used in combination. For
555 example, let's suppose we were interested in detecting targets between 2.0-2.5 m.
556 We could set just set _OffsetDistanceFromReference_ to 0.0 and "FrameStitch" to
557 3, but then much of the signal wouldn't be of any interest. In might be better in
558 this case to set _OffsetDistanceFromReference_ to 1.75 and _FrameStitch_ to 1.
559 _These settings should always allow some headroom in either direction since the
560 radar pulse itself spans a number of samples_.
561 
562 'SampleDelayToReference' is a delay (in nanoseconds) which is used to adjust
563 error in distance computations. There will be inherent delay due to internal
564 circuitry. There is also additional delay due to antenna cabling. This value
565 was determined empirically for the _Ancho_ cape, the antennas, and the 3" SMA
566 antenna cables. This value is different for the _Cayenne_ cape.
567 
568 The final three settings have a value "null." This indicates that if we want
569 to save the running JSON configuration, we are _interested_ in these values,
570 but are interpreted as read-only.
571 
572 Each of the BBB demo apps will include both stage1.json and stage2.json files.
573 However, some of the settings shown above may differ, and there also may be
574 additional settings. _Keep in mind that these files are simply a convenience to
575 avoid repeated replication of the code block shown at the beginning of this
576 section_. The "key" must be a string which represents a Radarlib3 variable. The
577 X1 and X2 share many variable names in common, but each also has its own unique
578 variables as well. See the Radarlib3 documentation.
